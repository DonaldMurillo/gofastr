# Task 038: CLI Build Command

**Phase:** 4 — CLI & DX  
**Depends on:** 035 (CLI Framework), 037 (Generate)  
**Status:** not started

---

## Goal

Implement `gofastr build` — the one-step production build command that runs code generation and then compiles the Go binary. This is the command developers and CI pipelines use to produce a deployable artifact.

---

## Context

From the proposal:

> `gofastr build` → codegen + go build → production binary

From the draft:

> **Compile-time checks** — entity declarations → codegen → `go build` catches everything

The build command orchestrates two steps:
1. **Generate** (task 037) — entity declarations → Go source code
2. **Compile** — `go build` the generated code + user code → single static binary

---

## Requirements

### 1. Command Definition

```
gofastr build [flags]
```

### 2. Flags

| Flag | Type | Default | Purpose |
|------|------|---------|---------|
| `-o` / `--output` | string | auto | Output binary path. Default: `./bin/<project-name>` (or `./bin/<project-name>.exe` on Windows). |
| `--os` | string | runtime GOOS | Target OS: `linux`, `darwin`, `windows`. |
| `--arch` | string | runtime GOARCH | Target architecture: `amd64`, `arm64`. |
| `--ldflags` | string | auto | Additional ldflags. Auto-includes version, commit, build date. |
| `--strip` | bool | true | Strip debug info (`-ldflags="-s -w"`). |
| `--trimpath` | bool | true | Remove local paths from binary (`-trimpath`). |
| `--no-generate` | bool | false | Skip code generation step (use existing `.gofastr/`). |
| `--race` | bool | false | Enable race detector (`-race`). |
| `--tags` | stringSlice | `[]` | Build tags to pass to `go build`. |

### 3. Build Pipeline

```
gofastr build
  │
  ├── Step 1: Generate (if not --no-generate)
  │   ├── Load config
  │   ├── Discover entities
  │   ├── Validate entities
  │   ├── Generate Go code to .gofastr/
  │   └── Report generated files
  │
  ├── Step 2: Version Injection
  │   └── Compute ldflags:
  │       -X main.version=<version>
  │       -X main.commit=<git-sha>
  │       -X main.buildDate=<timestamp>
  │
  ├── Step 3: Go Build
  │   ├── Set CGO_ENABLED (0 for static binary, 1 if sqlite)
  │   ├── Set GOOS / GOARCH
  │   ├── Run: go build -trimpath -ldflags="..." -o <output> .
  │   └── Capture stdout/stderr
  │
  └── Step 4: Report
      ├── Success: binary path, size, build time
      └── Failure: actionable error from codegen or go build
```

### 4. Version Injection

Default ldflags (merged with any user-provided `--ldflags`):

```
-X main.version=<version from config or git tag>
-X main.commit=<git rev-parse --short HEAD>
-X main.buildDate=<date -u '+%Y-%m-%dT%H:%M:%SZ'>
```

The `main.go` generated by `gofastr init` should have:

```go
var (
    version   = "dev"
    commit    = "none"
    buildDate = "unknown"
)
```

### 5. Cross-Compilation

When `--os` and `--arch` differ from the host:

```bash
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -trimpath -ldflags="..." -o bin/myapp .
```

For SQLite (which requires CGO), cross-compilation needs appropriate C cross-compilers. If `--db sqlite` is in config and `--os` differs from host, warn:

```
⚠ Cross-compiling with SQLite requires CGO and a C cross-compiler.
  Set CC=<cross-compiler> or use --db postgres for static cross-compilation.
```

### 6. Build Optimization

Default flags for production builds:

| Flag | Purpose |
|------|---------|
| `-trimpath` | Removes absolute paths from binary (reproducible builds, smaller) |
| `-ldflags="-s -w"` | Strip symbol table and DWARF debug info (smaller binary) |
| `CGO_ENABLED=0` | Static binary (no libc dependency). Only if not using SQLite. |

When `--strip=false`, omit `-s -w` to keep debug info for profiling.

### 7. Error Reporting

Both codegen errors and `go build` errors must be actionable:

#### Codegen errors (from generate step)

```
✗ Code generation failed:

  Entity 'posts' has an error:
    field 'author': relation target 'users' not found
    Available entities: posts, tags

  Fix: Create a 'users' entity or remove the 'author' field from 'posts'.
```

#### Go build errors

Parse `go build` output and enhance:

```
✗ Go build failed:

  .gofastr/entities/posts_routes.go:42: undefined: PostHandler

  This is likely a code generation issue.
  Try: gofastr build --clean to regenerate all files.
  If the issue persists, report a bug: https://github.com/gofastr/gofastr/issues
```

#### JSON error output

```json
{
  "status": "error",
  "error": "go build failed",
  "error_code": "BUILD_FAILED",
  "data": {
    "step": "compile",
    "output": ".gofastr/entities/posts_routes.go:42: undefined: PostHandler",
    "suggestions": [
      "Run 'gofastr build --clean' to regenerate all files",
      "Report a bug at https://github.com/gofastr/gofastr/issues"
    ]
  }
}
```

### 9. Build Caching

- If `.gofastr/` exists and no entity files changed since last generate, skip generation (unless `--clean`).
- Use file modification times for quick comparison.
- `--clean` forces full regeneration.
- Always run `go build` (Go's build cache handles this).

## Implementation Notes

- Use `os/exec.Command("go", "build", ...)` to invoke the Go toolchain. Capture both stdout and stderr.
- Parse `go build` error output to extract file:line:column for enhanced error messages.
- For version detection: try `git describe --tags --always --dirty` first, fall back to config version, then "dev".
- Time each step (generate, compile) separately for reporting.
- Binary size can be obtained with `os.Stat()` on the output file.
- The build command should work in CI (GitHub Actions, etc.) without requiring TTY — no interactive prompts, colors degrade gracefully.
