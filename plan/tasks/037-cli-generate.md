# Task 037: CLI Generate Command

**Phase:** 4 — CLI & DX  
**Depends on:** 035 (CLI Framework), 019 (Entity System)  
**Status:** not started

---

## Goal

Implement `gofastr generate` — the code generation command that transforms entity declarations (Go or JSON) and DSL queries into type-safe Go code. This is the core of GoFastr's compile-time safety guarantee: entity configs in → Go structs out → `go build` validates everything.

---

## Context

From the proposal:

> ```
> 1. Declare entities (Go or JSON)
> 2. gofastr generate → Go code (structs, routes, queries, MCP defs, OpenAPI)
> 3. go build → compiler validates EVERYTHING
> 4. Binary ships with zero "route not found" or "field doesn't exist" bugs
> ```

From the draft:

> Entity config → Go struct code (model types, input/output types)
> DSL queries → Go query struct code
> Template pre-compilation

The entity system (task 019) defines how entities are declared. This task is about the code generation pipeline that reads those declarations and produces Go source files.

## Requirements

### 1. Command Definition

```
gofastr generate [flags]
```

No positional arguments. Generates code based on the project's entity declarations and config.

### 2. Flags

| Flag | Type | Default | Purpose |
|------|------|---------|---------|
| `--dry-run` | bool | false | Show what would be generated without writing files. |
| `--watch` | bool | false | Watch for file changes and regenerate continuously. (Used by `gofastr dev`.) |
| `--clean` | bool | false | Delete `.gofastr/` before regenerating. |
| `--entities-dir` | string | `"entities/"` | Path to entities directory. |

### 3. Generation Pipeline

```
1. Load config (gofastr.yaml)
2. Discover entity declarations (Go files, JSON files, or directory scan)
3. Parse each entity into an intermediate representation (IR)
4. Validate all entities:
   - No duplicate names
   - All relations reference existing entities
   - All field types are valid
   - All required fields have names
5. Generate code from IR:
   a. Go structs (model, input, output types) per entity
   b. Query builder structs per entity
   c. Route registration code per entity (CRUD + custom endpoints)
   d. MCP tool definitions per entity
   e. OpenAPI spec fragments per entity
   f. Validation functions per entity
   g. Migration DDL per entity
6. Write generated files to .gofastr/
7. Report results
```

### 4. Output Directory: `.gofastr/`

All generated code goes into `.gofastr/` at the project root:

```
.gofastr/
├── generated.go              # single-file bundle (or split into files below)
├── entities/
│   ├── posts_model.go        # Post struct, PostInput, PostOutput
│   ├── posts_queries.go      # PostQuery struct, query builders
│   ├── posts_routes.go       # Route registration for posts entity
│   ├── posts_mcp.go          # MCP tool definitions for posts
│   ├── posts_openapi.go      # OpenAPI spec fragment
│   ├── posts_validate.go     # Validation functions
│   └── ...                   # one set per entity
├── routes.go                 # combined route registration
├── mcp.go                    # combined MCP tool registration
├── openapi.go                # combined OpenAPI spec
├── migrate.go                # migration DDL for all entities
└── templates.go              # pre-compiled templates (if applicable)
```

- `.gofastr/` is gitignored (regenerated on every build).
- Generated files have a header comment: `// Code generated by gofastr. DO NOT EDIT.`
- Each generated file includes the generation timestamp and gofastr version.

### 5. Generated Code: Entity Structs

For each entity, generate:

#### Model Type (database representation)

```go
// Code generated by gofastr v0.1.0. DO NOT EDIT.

package generated

import "time"

type Post struct {
    ID        string    `json:"id" db:"id"`
    Title     string    `json:"title" db:"title" validate:"required,max=200"`
    Body      string    `json:"body" db:"body"`
    Published bool      `json:"published" db:"published"`
    AuthorID  string    `json:"author_id" db:"author_id"`
    CreatedAt time.Time `json:"created_at" db:"created_at"`
    UpdatedAt time.Time `json:"updated_at" db:"updated_at"`
}
```

#### Input Type (create/update payloads)

```go
type PostCreateInput struct {
    Title     string `json:"title" validate:"required,max=200"`
    Body      string `json:"body"`
    Published *bool  `json:"published"` // pointer for optional bool
    AuthorID  string `json:"author_id"`
}

type PostUpdateInput struct {
    Title     *string `json:"title" validate:"omitempty,max=200"`
    Body      *string `json:"body"`
    Published *bool   `json:"published"`
    AuthorID  *string `json:"author_id"`
}
```

#### Output Type (API response)

```go
type PostOutput struct {
    ID        string     `json:"id"`
    Title     string     `json:"title"`
    Body      string     `json:"body"`
    Published bool       `json:"published"`
    Author    *UserOutput `json:"author,omitempty"` // if relation loaded
    CreatedAt time.Time  `json:"created_at"`
    UpdatedAt time.Time  `json:"updated_at"`
}
```

#### Conversion functions

```go
func (p *Post) ToOutput() PostOutput { ... }
func (i *PostCreateInput) ToModel() Post { ... }
func ApplyUpdate(model *Post, input *PostUpdateInput) { ... }
```

### 11. Watch Mode (`--watch`)

- Uses `fsnotify` to watch for file changes.
- Watch paths: `entities/`, `templates/`, config file.
- Ignore: `.gofastr/`, `.git/`, `vendor/`, `node_modules/`.
- On file change: debounce 200ms, then regenerate.
- Print change events in verbose mode.
- This is the foundation for `gofastr dev` (task 039), which adds rebuild + restart on top.

### 12. Template Pre-Compilation

If the project has templates (HTML templates for the rendering engine):
- Scan `templates/` directory.
- Pre-compile templates into Go code.
- Generate `templates.go` with compiled template functions.

## Intermediate Representation (IR)

The generator needs an intermediate representation that both Go and JSON entity declarations compile to:

```go
type EntityIR struct {
    Name        string
    Table       string
    Fields      []FieldIR
    Relations   []RelationIR
    CRUD        bool
    MCP         bool
    SoftDelete  bool
    MultiTenant bool
    Endpoints   []EndpointIR
    Validators  []ValidatorIR
    Hooks       HooksIR
    Access      AccessIR
}

type FieldIR struct {
    Name     string
    Type     FieldType  // String, Text, Int, Float, Decimal, Bool, Enum, Date, etc.
    Required bool
    Unique   bool
    Default  interface{}
    Max      int
    Min      int
    Values   []string   // for Enum
    Relation *RelationIR
}

type RelationIR struct {
    Entity   string
    Many     bool
    Optional bool
}
```

The entity system (task 019) should provide the parser that converts Go/JSON declarations → `EntityIR`. This task uses the IR to generate code.

## Error Handling

| Error | Message | Suggestion |
|-------|---------|------------|
| No entities found | `No entity declarations found in entities/.` | `Create an entity file in entities/ or specify a different path with --entities-dir.` |
| Invalid entity JSON | `Parse error in entities/posts.json: line 12: unknown field type "xyz"` | `Valid types: string, text, int, float, decimal, bool, enum, date, relation, image, file` |
| Circular relation | `Circular relation detected: posts.author → users.favorite_post → posts` | `Remove one side of the circular relation or use a junction table.` |
| Missing relation target | `Entity 'posts' references 'comments' but no 'comments' entity exists.` | `Create a 'comments' entity or fix the relation field.` |
| Duplicate entity name | `Entity 'posts' is defined in both entities/posts.json and entities/posts_v2.json.` | `Remove or rename one of the entity declarations.` |

## Implementation Notes

- Use Go's `text/template` for code generation templates. Consider string-based templates for simplicity.
- The IR types should live in a shared package (e.g., `gofastr/internal/generate/ir/`) that both the entity parser and the code generator can import.
- Code generation must be deterministic: same input → same output, regardless of map iteration order. Sort entity names, field names, etc.
- For the watch mode, use `fsnotify/fsnotify` — same library that `gofastr dev` will use.
- Consider adding `//line` directives in generated code pointing to the source entity file for better error messages during `go build`.
