# Task 035: CLI Framework

**Phase:** 4 — CLI & DX  
**Depends on:** nothing  
**Status:** not started

---

## Goal

Build the foundational CLI framework for GoFastr using Cobra. This is the `gofastr` root command that all other CLI tasks (init, generate, build, dev, migrate, test) register into. It must support both human-friendly colored output and machine-readable JSON output for AI agents.

---

## Context

GoFastr is a Go fullstack framework with a two-layer architecture (`core/` primitives + `framework/` entity system). The CLI is the primary interface for developers and AI coding agents. From the draft:

> ```
> gofastr init "blog with posts and tags"   → scaffold project + entities
> gofastr dev                               → hot-reload: watch → generate → build → restart
> gofastr build                             → codegen + go build → production binary
> gofastr migrate                           → run pending migrations
> gofastr test                              → run tests with framework helpers
> ```
>
> All commands support `--json` for AI agent consumption.

The CLI also needs to support the plugin system (`CLIPlugin` interface: `AddCommands(root *cobra.Command)`).

---

## Requirements

### 1. Root Command: `gofastr`

- Create a `cmd/gofastr/` package (or `cli/` — choose convention and be consistent).
- Root command is `gofastr` with no action (prints help).
- Use [spf13/cobra](https://github.com/spf13/cobra) as the CLI framework.
- Use [fatih/color](https://github.com/fatih/color) or [charmbracelet/lipgloss](https://github.com/charmbracelet/lipgloss) for colored output.

### 2. Global Flags

| Flag | Type | Default | Purpose |
|------|------|---------|---------|
| `--json` | bool | false | Machine-readable JSON output for AI agents. Suppresses all color and interactive prompts. |
| `--verbose` / `-v` | bool | false | Verbose logging: show internal steps, timing, file paths. |
| `--config` | string | `""` | Path to config file. If empty, auto-discover `gofastr.yaml` or `gofastr.json` in CWD and parents. |

### 3. Config File Loading

- Search order: explicit `--config` path → `gofastr.yaml` in CWD → `gofastr.json` in CWD → walk up to repo root.
- Config struct (initial version, extendable):

```go
type Config struct {
    Name    string            `yaml:"name" json:"name"`
    Version string            `yaml:"version" json:"version"`
    DB      DatabaseConfig    `yaml:"db" json:"db"`
    Server  ServerConfig      `yaml:"server" json:"server"`
    Dev     DevConfig         `yaml:"dev" json:"dev"`
}

type DatabaseConfig struct {
    Driver   string `yaml:"driver" json:"driver"`     // "postgres", "sqlite", etc.
    URL      string `yaml:"url" json:"url"`
    MaxConns int    `yaml:"max_conns" json:"max_conns"`
}

type ServerConfig struct {
    Port int    `yaml:"port" json:"port"`
    Host string `yaml:"host" json:"host"`
}

type DevConfig struct {
    Port       int      `yaml:"port" json:"port"`
    OpenBrowser bool    `yaml:"open_browser" json:"open_browser"`
    WatchPaths []string `yaml:"watch_paths" json:"watch_paths"`
    IgnorePaths []string `yaml:"ignore_paths" json:"ignore_paths"`
}
```

- Store loaded config in a context or package-level var accessible to all sub-commands.
- If no config file found, sub-commands that require one should print an actionable error: `"No gofastr config found. Run 'gofastr init' to create a project, or use --config to specify a path."`

### 4. Sub-Command Registration Pattern

- Each sub-command (init, generate, build, dev, migrate, test) registers itself via an init function or explicit registration call.
- Pattern must be extensible by plugins implementing `CLIPlugin`:

```go
// From the plugin system in draft.md:
type CLIPlugin interface {
    AddCommands(root *cobra.Command)
}
```

- Suggested pattern: each command lives in its own file (`cmd_init.go`, `cmd_build.go`, etc.) within the CLI package, with a `Register(root *cobra.Command)` function.

### 5. Output System: Dual-Mode (Human vs Machine)

Create an `output` package or utility:

```go
type Output struct {
    json     bool
    verbose  bool
    printer  *color.Color
}

// Human mode: colored, indented, with icons (✓ ✗ →)
func (o *Output) Success(msg string, args ...interface{})
func (o *Output) Error(msg string, args ...interface{})
func (o *Output) Info(msg string, args ...interface{})
func (o *Output) Step(msg string, args ...interface{})   // verbose only
func (o *Output) Table(headers []string, rows [][]string)

// Machine mode (--json): structured JSON to stdout
func (o *Output) JSON(data interface{})
func (o *Output) JSONError(err error, suggestions []string)
```

When `--json` is set:
- ALL output goes to stdout as structured JSON.
- Format: `{"status": "success"|"error", "data": ..., "error": "...", "suggestions": [...]}`.
- No color codes, no interactive prompts, no spinner animations.

### 6. `gofastr version` Command

- Prints version, commit hash, build date.
- Version is injected at build time via `-ldflags`.
- Human: `GoFastr v0.1.0 (abc1234, 2025-01-15)`.
- JSON: `{"version": "0.1.0", "commit": "abc1234", "build_date": "2025-01-15"}`.

### 7. Help Generation

- Cobra's default help, customized with:
  - GoFastr branding in the header.
  - Examples section for each command.
  - "See also" links between related commands.
- `gofastr help [command]` works as expected.
- `--help` / `-h` on any command.

### 8. Error Handling

All errors must be actionable:

```go
// Bad:  fmt.Println("error: file not found")
// Good:
o.Error("Config file not found: %s", path)
o.Info("Suggestions:")
o.Info("  • Run 'gofastr init' to create a new project")
o.Info("  • Use --config to specify a different path")
o.Info("  • Make sure you're in a GoFastr project directory")
```

Error categories to handle:
- **No config found** → suggest `gofastr init` or `--config`
- **Not in a project directory** → suggest `cd` to project or `gofastr init`
- **Database connection failed** → show connection string (mask password), suggest checking config
- **Unknown command** → suggest similar commands (fuzzy match)

For `--json` mode, errors are structured:

```json
{
  "status": "error",
  "error": "config file not found",
  "error_code": "CONFIG_NOT_FOUND",
  "suggestions": [
    "Run 'gofastr init' to create a new project",
    "Use --config to specify a path"
  ]
}
```

## Implementation Notes

- Use `go mod init github.com/user/gofastr` (or whatever the module path is) if not already done.
- Pin Cobra to v1.x. Use `cobra-cli` scaffold if helpful, but strip it down — we don't need viper for config (just YAML/JSON loading).
- For YAML config: use `gopkg.in/yaml.v3`. For JSON: stdlib `encoding/json`.
- The `Output` type should be created once at startup from global flags and passed to all commands.
- Consider using `context.Context` to carry both the config and output mode through the command tree.
- Keep the CLI package importable so that the test harness (task 041) can invoke commands programmatically.
